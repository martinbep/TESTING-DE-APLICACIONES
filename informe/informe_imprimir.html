Parte 1 - Diseño del escenario de prueba

Aplicación elegida

Para este TP armé una calculadora simple en Python con cuatro operaciones básicas: sumar, restar, multiplicar y dividir. La lógica se escribió buscando que sea fácil de entender y, sobre todo, que se pueda probar de forma clara. La elección del dominio (una calculadora) permite cubrir los tres tipos de casos pedidos en la consigna sin complicar el código: hay caminos felices (operaciones válidas), errores controlados (división por cero, parámetros inválidos) y casos borde (números muy grandes, negativos, ceros).

Aunque la app es simple, se respetó la estructura típica de un proyecto Python con responsabilidades separadas:

  • app/calculadora.py: contiene la lógica de negocio (las cuatro funciones).
  • main.py: punto de entrada, ejecuta una pequeña demo por consola.
  • tests/test_calculadora.py: tests automatizados con pytest.
  • resources/: carpeta para recursos (capturas, datos de prueba, etc.).
  • .github/workflows/tests.yml: definición del pipeline en GitHub Actions.

Escenario de prueba

Contexto: un usuario utiliza las funciones de la calculadora para realizar operaciones aritméticas. La calculadora debe devolver el resultado correcto cuando los datos son válidos, lanzar errores claros cuando los datos no lo son, y comportarse de forma predecible en situaciones límite.

Parte 2 - Automatización de pruebas

Los casos diseñados arriba se automatizaron en Python usando pytest, que es la herramienta estándar del ecosistema y la que recomienda Federico Toledo en el capítulo de automatización de pruebas funcionales. Pytest tiene una sintaxis liviana (un test es simplemente una función que empieza con test_ y usa assert) y se integra muy fácil con cualquier pipeline CI/CD.

Tests implementados

Se escribieron 8 tests automatizados, agrupados por tipo (exitoso, error, borde). El extracto más representativo es el siguiente:

import pytest
from app.calculadora import sumar, dividir, multiplicar, restar

# 1) CASO EXITOSO
def test_sumar_dos_enteros_positivos():
    assert sumar(2, 3) == 5

# 2) CASO DE ERROR
def test_dividir_por_cero_lanza_value_error():
    with pytest.raises(ValueError) as exc_info:
        dividir(10, 0)
    assert 'cero' in str(exc_info.value).lower()

# 3) EDGE CASE
def test_sumar_numeros_muy_grandes():
    grande = 10**18
    assert sumar(grande, grande) == 2 * grande

El test de división por cero usa pytest.raises, que verifica que el código lance la excepción esperada. Si la función no lanza ese error, el test falla. Es la forma estándar de testear caminos de error sin que el test mismo se rompa.

Parte 3 - Implementación del pipeline (CI/CD)

El pipeline está definido como un workflow de GitHub Actions en el archivo .github/workflows/tests.yml. Cada vez que se hace push a la rama main, GitHub levanta una máquina virtual con Ubuntu, clona el repo, instala Python, instala las dependencias del proyecto, corre los tests y, al final, sube el reporte HTML como un artefacto descargable.

Definición del workflow

name: CI - Tests Calculadora

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - name: 1. Clonar el repositorio
        uses: actions/checkout@v4

      - name: 2. Configurar Python 3.11
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: 3. Instalar dependencias
        run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt

      - name: 4. Ejecutar tests con pytest y generar reporte HTML
        run: |
          pytest --html=report.html --self-contained-html

      - name: 5. Subir el reporte HTML como artefacto
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: reporte-pytest
          path: report.html

Qué hace cada paso

  • checkout@v4: clona el repo dentro del runner de GitHub.
  • setup-python@v5: instala Python 3.11 en la máquina virtual.
  • pip install -r requirements.txt: instala pytest y pytest-html.
  • pytest --html=report.html: corre los tests y genera el reporte.
  • upload-artifact@v4: deja el reporte disponible para descargar desde la UI de Actions, incluso si los tests fallaran (por if: always()).

Breve explicación de los resultados

Los 8 tests pasaron en menos de un segundo. Los casos exitosos confirman que las cuatro operaciones devuelven el resultado correcto. Los casos de error confirman que la calculadora no se cuelga ante datos inválidos sino que lanza excepciones claras (ValueError para división por cero, TypeError para parámetros no numéricos). Los edge cases verifican que el comportamiento es estable con negativos, ceros y números muy grandes.

Preguntas

¿Qué ventaja aporta automatizar pruebas dentro del pipeline?

La principal ventaja es que los tests corren solos cada vez que alguien sube un cambio. Así nadie se "olvida" de ejecutarlos antes de mergear y los errores se detectan en el momento mismo en que se introducen, no varios días después cuando ya está todo enredado. También sirve como red de seguridad: si yo modifico la función dividir y rompo sin querer otra parte, el pipeline lo marca al toque.

¿Qué impacto tiene esto en la calidad del software?

Mejora la calidad porque baja los tiempos de respuesta ante un bug y obliga a mantener el código testeable. Como cada commit tiene que pasar por el pipeline para considerarse válido, se evita que código roto llegue a la rama principal. También empuja al equipo a escribir tests para las funcionalidades nuevas, lo que termina dando una base de pruebas que crece con el proyecto.

Parte 5 - Reflexión

¿Qué beneficios tiene automatizar pruebas frente a hacerlas manualmente?

Lo que más se nota es la velocidad y la repetibilidad. Probar a mano una calculadora con 8 casos lleva varios minutos; con pytest tarda menos de un segundo y siempre se ejecuta exactamente igual. Otro punto fuerte es que se puede correr cada vez que se modifica algo, sin pereza ni descuidos. Las pruebas manuales sirven para exploración y usabilidad, pero para regresión no hay con que darle a la automatización.

¿Qué dificultades encontré al implementar el pipeline?

Lo más confuso fue entender la estructura del archivo YAML de GitHub Actions, sobre todo la indentación (un espacio mal puesto rompe todo el workflow). También me costó entender que actions/checkout no es magia, sino una acción pública que hay que invocar explícitamente para que el runner pueda ver el código. Una vez que entendí que el pipeline es básicamente un script que se ejecuta paso a paso en una máquina limpia, todo cerró más fácil.

¿Dónde aplicarías este enfoque en un entorno real?

En cualquier proyecto que tenga más de una persona o que se actualice seguido. Lo aplicaría para correr tests unitarios en cada PR, validar que el código cumpla con un linter, correr tests de integración contra una base de datos efímera y, eventualmente, automatizar el deploy a un entorno de staging si los tests pasan. En proyectos chicos y personales puede parecer "mucha ceremonia", pero apenas el repo crece y participan dos o más personas, tener un pipeline andando deja de ser opcional.

Conclusiones personales

Hacer este TP me sirvió para aterrizar conceptos que veníamos viendo en teoría sobre automatización. Una cosa es leer que los tests automatizados ahorran tiempo, y otra es ver el tilde verde aparecer solo en GitHub después de un push. La parte que más me gustó fue darme cuenta de que el pipeline no es algo enorme: son cinco pasos en un archivo YAML, cada uno con un objetivo claro. Sumando la calculadora, los tests y el pipeline, en el repo quedan menos de 200 líneas de código, pero el flujo cubre todo el ciclo básico de DevOps que arranca cuando alguien hace push.

Propuestas de futuras mejoras

  • Sumar cobertura de código con pytest-cov para medir qué porcentaje del código está cubierto por tests y exigir un mínimo (por ejemplo, 80%) en el pipeline.
  • Agregar un linter (flake8 o ruff) como step previo al pytest, así también se controla el estilo del código.
  • Probar la app en varias versiones de Python (3.10, 3.11, 3.12) usando una matriz en el workflow, para detectar incompatibilidades.
  • Sumar badges en el README mostrando el estado del pipeline y la cobertura, para que se vea de un solo vistazo.
  • Cuando la app crezca, separar tests unitarios (rápidos) de tests de integración (más lentos) y configurar dos jobs en paralelo.
